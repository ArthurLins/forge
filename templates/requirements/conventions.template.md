# {{PROJECT_NAME}} — Conventions Map (`EC`)

> **Purpose:** the project's registry of **cross-cutting engineering & UX
> defaults** that every feature must honor — pagination, list virtualization,
> async loading/empty/error states, debounced search, avoid N+1, accessibility,
> authorization on every action, and so on. It exists so neither the developer
> nor the agent has to remember these on each feature: the conventions are
> **recalled** here and **enforced** automatically (see "How this map is
> enforced" below).
>
> This map is **generated by [`/forge-init`](../../.claude/skills/forge-init/SKILL.md)**
> from [`templates/conventions-catalog.md`](../../templates/conventions-catalog.md)
> (scaled to the project type and size) and **grown via
> [`/forge-add-convention`](../../.claude/commands/forge-add-convention.md)**.

**ID convention:** engineering conventions use the prefix **`EC`** (e.g.
`EC-01`), zero-padded and **never renumbered**; retire an entry by setting its
Status to `waived` with a reason rather than reusing its number. Optionally tag
honoring code/tests with `@convention EC-01` for lightweight traceability (see
the [index](index.md) taxonomy). `EC` differs from `NFR` (project-wide quality
targets) and from `standards.md` (code style): an `EC` is an **applied,
cross-cutting default** a feature must honor, matched by its **Applies to**
scope and checked by the reviewer.

---

## EC entry format

Every entry has exactly these fields:

- **Category** — the area it belongs to (Collections · Forms · Data fetching ·
  Async UI states · Performance · Navigation · Accessibility · Security ·
  Internationalization · Observability · …).
- **Rule (intent)** — the convention as a **stack-neutral intent** (*what*, never
  *with which tool*).
- **Applies to** — the trigger scope: the kind of work that makes this
  convention relevant. The feature prompts and the reviewer match against this.
- **Parameters** — named thresholds with **values** (defaulted from the catalog,
  editable per project).
- **Rationale** — why it is worth honoring.
- **Status** — `active` | `proposed` | `waived` (with a reason). Only `active`
  entries are enforced; `proposed` ones await the developer's approval; `waived`
  ones are intentionally not applied and say why.

## How this map is enforced (two teeth)

1. **Feature prompts read it.** Every feature-building prompt is instructed to
   read this file, find each `EC-` whose **Applies to** matches its work, apply
   it within its **Parameters**, and record which `EC` ids it honored (in code
   and tests, optionally as `@convention EC-xx`).
2. **The reviewer checks it.** The
   [`reviewer`](../../.claude/agents/reviewer.md) subagent loads this map and,
   for the changed code, **rejects** when an applicable **active** `EC` was
   ignored or violated. (No conventions map, or no matching `EC` → the check is
   skipped and reported as such.)

> There is **no separate Definition-of-Done checklist gate** and **no
> planning-time mapping step** — these two teeth are the whole enforcement.

## Growing the map

Add or amend an entry with
[`/forge-add-convention`](../../.claude/commands/forge-add-convention.md): pick
one from the [catalog](../../templates/conventions-catalog.md) or define a custom
one, set its **Applies to** and **Parameters**, keep ids sequential, and mark its
**Status**. Agents may also **propose** a new `EC` when they notice a recurring
concern while building — recorded through the same command as a `proposed` entry
for the developer to approve.

### EC-01 — Paginate or window every unbounded collection

- **Category:** Collections, lists & tables
- **Rule (intent):** Never render or return an unbounded list. Any view or
  response over a collection that can grow is paginated (or virtualized/windowed
  when shown as one continuous scroll), so cost stays bounded regardless of total
  size.
- **Applies to:** any screen, endpoint, export, or command that lists items from
  a collection that can exceed the page threshold.
- **Parameters:** `defaultPageSize = 25`; `maxPageSize = 100`;
  `windowThreshold = 100`.
- **Rationale:** unbounded lists are the most common cause of slow pages, huge
  payloads, and memory blow-ups; bounding them is cheap insurance.
- **Status:** active

> *Proposed/waived examples (project decides):* an entry the agent suggested mid-
> build would sit here as `proposed` until approved; one the team chose not to
> apply would be `waived` with the reason — both omitted above on purpose.
